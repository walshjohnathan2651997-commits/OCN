# Dataset Shortcut Risk Assessment V1

## Purpose

This document explains why the 444-candidate controlled silver diagnostic
set is **not waste data**, why it **cannot carry benchmark claims**, and
why it **can** be used as a retrieval-to-screening diagnostic.

It is intended to accompany the leakage audit results in
`experiments/leakage_audit_v1/audit_summary.md` and to provide reviewers
with the context needed to interpret R4 screening results honestly.

## What the 444 Dataset Is

The 444-candidate set is a **controlled silver diagnostic** constructed
from local PDFs via the SimClaim generation pipeline. Each candidate
has:

- A `claim_text` generated from evidence in a local PDF
- A `candidate_label_guess` (silver label) assigned by the generation
  policy, NOT by human annotators
- A `target_candidate_group_id` linking related candidates (4 per group)
- A `claim_generation_policy` recording how the candidate was generated
- A `domain` (autonomous_driving, policy_simulation, digital_twin,
  cyber_defense, marl, robotics)

The 4 labels are: `supported`, `mild_scope_overclaim`,
`strong_action_overclaim`, `contradiction_candidate` (111 each, balanced).

**Crucially**: `human_audited=False` for all 444 candidates, and
`final_label` / `gold_label` are empty. The labels are silver
(machine-assigned), not gold (human-verified).

## Why 444 Is Not Waste Data

Despite being silver-labeled, the 444 set has diagnostic value:

1. **Balanced label coverage**: 111 candidates per label, allowing
   meaningful per-label metric computation without prevalence distortion.

2. **Group structure**: 111 groups of 4 candidates each, enabling
   group-aware cross-validation that prevents leakage of related
   candidates between train and test folds.

3. **Multi-domain coverage**: 6 domains (autonomous_driving, policy_simulation,
   digital_twin, cyber_defense, marl, robotics), allowing domain-stratified
   analysis.

4. **Complete source trace**: All 444 candidates have
   `source_trace_complete=True`, meaning the evidence PDF, page, and
   span are traceable. This allows retrieval and canonicalization to be
   evaluated end-to-end.

5. **Leakage audit PASS**: The enhanced leakage audit (12 checks) found
   no significant shortcuts:
   - Claim-only TF-IDF+LR strong_F1=0.2448 (54% of R4's 0.4503) — R4
     uses evidence beyond claim text
   - Correct evidence overlap=0.36 vs shuffled=0.008 (46x ratio) —
     pipeline is evidence-sensitive
   - Policy-label concentration=0.25 (perfectly balanced) — no
     policy-label shortcut
   - Cue word rates similar across labels (max ratio 1.6) — no
     lexical cue shortcut
   - 0 cross-split groups — no group leakage

## Why 444 Cannot Carry Benchmark Claims

Despite passing the leakage audit, the 444 set **cannot** be used as a
benchmark for the following reasons:

1. **Silver labels, not gold**: `candidate_label_guess` is assigned by
   the generation policy, not by human annotators. Label noise is
   unknown but non-zero. Any metric computed against these labels is
   an upper bound on performance against gold labels.

2. **No human audit**: `human_audited=False` for all 444. Without human
   verification, we cannot claim the labels are correct, only that they
   are internally consistent with the generation policy.

3. **Controlled generation, not natural distribution**: Candidates were
   generated from PDFs using specific policies
   (`simclaim_ai_preannotated_candidates_no_gold_no_final_label`,
   `simclaim_four_variant_v1`, etc.). The label distribution is
   artificially balanced (111 each), not reflecting natural prevalence.
   Metrics like precision, recall, and F1 are computed on a balanced
   set and do not generalize to real-world deployment.

4. **Generation-policy artifacts**: Even though the leakage audit found
   no strong shortcuts, the claims were generated by GPT/silver
   policies. Residual generation traces (phrasing patterns, vocabulary
   preferences) may inflate metrics relative to naturally-written
   claims. The claim-only baseline (strong_F1=0.2448) confirms some
   signal exists in claim text alone, even if it's below the warning
   threshold.

5. **No external validation**: The 444 set has not been compared to any
   external gold benchmark (e.g., SciFact, VitaminC). Without external
   validation, we cannot claim the screening model generalizes to
   real scientific claims.

## Why 444 Can Be Used for Retrieval-to-Screening Diagnostic

The 444 set is appropriate for **diagnostic** purposes:

1. **End-to-end pipeline validation**: The set allows testing whether
   the full pipeline (PDF → BM25 retrieval → canonicalization → R4
   screening → SmartQueue) runs end-to-end without errors. This is
   an engineering validation, not a performance claim.

2. **Relative comparison of selectors**: The canonicalizer ablation
   compares 8 evidence selectors on the same 444 candidates. Relative
   improvements (e.g., `best_sentence_top5_overlap` vs `raw_top1_chunk`)
   are meaningful even with silver labels, because all selectors are
   evaluated against the same label noise.

3. **Leakage detection**: The set allows running the 12-check leakage
   audit to verify the pipeline does not cheat (no oracle in selector,
   no label in queue sorting, no claim-only shortcut). This is a
   safety check, not a performance claim.

4. **Evidence sensitivity verification**: The shuffled-evidence baseline
   (correct overlap=0.36 vs shuffled=0.008) confirms the pipeline is
   actually using evidence, not just claim text. This is a sanity check
   on the pipeline's evidence dependence.

5. **Controlled experimentation**: The balanced label distribution and
   group structure allow controlled experiments (e.g., "does selector X
   improve recall over selector Y?") without confounding by prevalence.
   Results should be reported as relative comparisons, not absolute
   performance.

## Safe vs Unsafe Claims

### Safe Claims (allowed)

- "On a controlled silver diagnostic set of 444 candidates..."
- "Silver-labeled retrieval-to-screening diagnostic..."
- "Relative improvement of selector X over selector Y..."
- "Pipeline runs end-to-end without leakage..."
- "Evidence sensitivity confirmed (correct > shuffled)..."
- "Not a gold benchmark; labels are silver..."

### Unsafe Claims (forbidden)

- "Gold benchmark performance..."
- "Validated general detector..."
- "Human-audited results..."
- "SOTA performance..."
- "Generalizes to real scientific claims..."
- "Production-ready screening..."

## Interpretation for Paper

When reporting R4 screening results on the 444 set:

1. **Always qualify**: "controlled silver diagnostic set (N=444,
   silver-labeled, not human-audited)"

2. **Report relative comparisons**: "selector X improves oracle recall
   by 9x over selector Y" — not "selector X achieves 38.7% recall"

3. **Cite the leakage audit**: "12-check leakage audit found no
   significant shortcuts (claim-only strong_F1=0.24, evidence
   sensitivity ratio=46x)"

4. **Disclaim generalization**: "Results are diagnostic and do not
   generalize to real scientific claims without human audit and
   external validation"

5. **No SOTA claim**: "This is not a SOTA result; it is a controlled
   diagnostic on a silver-labeled set"
